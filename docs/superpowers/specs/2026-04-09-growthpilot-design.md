# GrowthPilot 设计文档

## 1. 项目概述

`GrowthPilot` 是一个面向中国大陆大学生的中文成长管理产品，核心目标不是简单地“记录任务”，而是通过 `目标拆解 -> 每日执行 -> 行为记录 -> 周期复盘 -> 数据反馈` 的闭环，帮助用户更稳定地推进学习、自律与求职相关目标。

产品的核心价值主张是：

- 帮用户把模糊目标转成可执行动作
- 帮用户理解自己为什么坚持不下来
- 帮用户通过数据反馈持续调整策略

它应当同时满足以下定位：

- 是一个真实可交互、可部署的全栈产品
- 能体现产品经理在用户洞察、功能设计、数据分析、指标体系、MVP 取舍上的能力
- 具备明显的审美和品牌气质，不呈现为“普通效率工具”

## 2. 目标与非目标

### 2.1 项目目标

- 在两周内完成一个可交互、可演示的作品集级产品
- 清晰服务于中国大陆大学生的成长执行问题
- 具备完整前端、后端、数据库与 AI 增强能力
- 具备 PM 数据后台，可展示指标、漏斗、分群与行为洞察
- 支持未来面向中国大陆用户的公开访问，优先选用有免费额度的平台和工具

### 2.2 非目标

- 第一版不做复杂社交关系链
- 第一版不做重内容社区
- 第一版不做复杂商业化与支付
- 第一版不做复杂权限和多组织系统
- 第一版不做“全场景万能平台”，只围绕成长执行闭环做深

## 3. 目标用户

### 3.1 核心用户

中国大陆 18-24 岁大学生，典型特征为：

- 同时面对学习、自律、求职等多个目标
- 主观上想努力，但执行不稳定
- 会计划，但容易拖延、反复中断、缺少复盘
- 愿意接受温和的陪伴和反馈，不喜欢高压式管理

### 3.2 典型场景优先级

优先级按照用户确认结果排序如下：

1. 综合自律与成长管理
2. 学习备考
3. 求职准备

### 3.3 核心用户问题

- 目标太大，不知道如何开始
- 每天任务很多，不知道什么最重要
- 容易拖延，但说不清自己为什么拖延
- 即使做了计划，也缺少阶段性复盘
- 没有一个真正“看懂我行为模式”的工具

## 4. 产品定位

`GrowthPilot` 不是待办清单，也不是纯打卡产品。它是一个“行动数据反馈产品”。

相对于普通任务管理工具，它的差异点在于：

- 重点不是记录任务，而是帮助用户理解行动模式
- 重点不是强提醒，而是温和的反馈和策略调整
- 重点不是堆叠功能，而是把一个行为闭环做深
- 重点不是后台感，而是用柔和、愉悦、灵动的视觉降低使用压力

一句话定位：

> 用中文、轻盈且富有情绪价值的交互方式，帮助中国大陆大学生把长期目标拆成可持续的小行动，并通过数据反馈持续成长。

## 5. 核心体验原则

### 5.1 中文优先

- 用户端核心文案全部以中文为主
- 英文仅保留项目品牌名 `GrowthPilot` 和少量开发层命名
- 文案风格要温和、克制、有陪伴感，不使用冰冷命令式表达

### 5.2 柔和、轻盈、灵动

视觉与交互应参考用户偏好的风格，整体气质为：

- 柔和：低饱和自然色、奶油白、浅金、鼠尾草绿、雾蓝
- 轻盈：大留白、朦胧渐变、半透明层、轻玻璃拟态
- 灵动：微视差、漂浮粒子、图表生长、卡片呼吸感
- 愉悦：关键反馈有温度、有情绪价值，不做生硬的成败提示

### 5.3 数据反馈而非数据压迫

- 图表和指标是为了帮助用户理解自己，而不是制造焦虑
- 周复盘应给出洞察与建议，而不是只有“完成/未完成”二元评判
- 当用户掉线时，产品提供低压的重新开始机制

### 5.4 高频页优先

第一版优先把用户高频和最有差异化的页面做好：

- 首页/落地页
- 成长驾驶舱
- 今日行动页
- 周复盘页
- PM 数据后台

## 6. 核心用户旅程

GrowthPilot 的主闭环如下：

1. 用户设定一个成长目标
2. 系统用 AI 或规则模板进行目标拆解
3. 用户每天完成少量关键动作
4. 系统记录执行结果、耗时、拖延原因与状态
5. 周期性生成复盘与洞察
6. 用户基于反馈微调下一阶段策略

这个闭环对应产品的核心价值：

- 降低开始难度
- 提升持续行动概率
- 沉淀行为数据
- 输出可解释的自我反馈

## 7. 信息架构

### 7.1 用户端一级页面

#### 7.1.1 首页 Landing

职责：

- 传达品牌感和情绪氛围
- 让用户第一眼理解产品价值
- 展示柔和、轻盈、灵动的视觉风格

#### 7.1.2 首次引导 Onboarding

职责：

- 收集用户当前目标、场景偏好、节奏偏好
- 为目标拆解提供结构化输入

#### 7.1.3 成长驾驶舱 Dashboard

职责：

- 展示今日关键任务、连续行动天数、风险提醒、本周一句洞察
- 承接用户最高频访问

#### 7.1.4 目标页 Goals

职责：

- 展示长期目标、阶段里程碑、AI 拆解结果与任务结构
- 支持重新规划

#### 7.1.5 今日行动页 Focus

职责：

- 承接用户真正开始行动的时刻
- 记录任务开始、完成、中断与原因

#### 7.1.6 周复盘页 Review

职责：

- 展示完成率、拖延原因分布、最佳执行时段、AI 洞察与下周建议

#### 7.1.7 个人页 Profile

职责：

- 展示成长标签、偏好设置、徽章、产品偏好
- 第一版可轻量实现，不作为和驾驶舱、行动页、复盘页同级的优先开发重点
### 7.2 PM 数据后台 Console

后台独立为一套内部视图，用于体现 PM 视角能力，最少包含：

- 核心指标总览
- 行为漏斗
- 用户分群
- 任务执行趋势
- 拖延原因分析
- 周复盘使用情况

## 8. 核心功能模块

### 8.1 目标设定 Goal Setup

用户输入：

- 当前主要目标
- 目标分类：自律 / 学习 / 求职
- 截止时间
- 计划强度
- 每周可投入时间

设计要求：

- 输入过程简洁，不堆砌表单
- 文案要让用户感觉在“整理生活”，不是“填系统”

### 8.2 计划拆解 AI Planner

系统根据目标生成：

- 阶段目标
- 本周重点
- 今日最小行动

设计要求：

- 输出结构稳定，便于前端展示
- AI 失败时回退到规则模板
- 每次拆解结果支持用户手动编辑

### 8.3 每日执行 Daily Focus

支持用户：

- 开始任务
- 完成任务
- 暂停或放弃任务
- 记录拖延或中断原因
- 快速记录精力或状态

设计要求：

- 页面必须足够轻和专注，不要像后台管理界面
- 完成反馈有柔和动态效果

### 8.4 周复盘 Review & Insights

系统自动汇总：

- 本周任务完成率
- 目标推进情况
- 高频拖延原因
- 高效执行时段
- AI 生成的中文复盘总结
- 下周行动建议

设计要求：

- 重点突出“洞察”，不是简单堆图表
- 复盘文案有陪伴感和可执行性

### 8.5 轻游戏化激励

作为辅助机制，仅保留轻量元素：

- 连续行动天数
- 徽章
- 成长等级

不作为核心驱动力，避免产品变成纯打卡游戏。

### 8.6 PM 数据后台

支持展示：

- 目标创建率
- AI 计划生成率
- 首次行动启动率
- 周复盘完成率
- 次日留存和 7 日留存
- 用户分群行为差异

## 10. 指标体系

### 10.1 北极星指标

第一版采用双核心衡量：

- 周有效行动次数
- 周复盘完成率

原因：

- 该产品的价值不在单日活跃，而在持续行动和闭环复盘

### 10.2 关键漏斗

核心漏斗定义为：

`创建目标 -> 生成计划 -> 首次开始任务 -> 首次完成任务 -> 完成周复盘`

### 10.3 核心业务指标

增长指标：

- 目标创建率
- AI 计划生成率
- 首次任务启动率

留存指标：

- 次日留存
- 7 日留存
- 周复盘回访率

执行指标：

- 任务完成率
- 任务放弃率
- 重规划率
- 拖延率

洞察指标：

- 拖延原因分布
- 目标类型完成差异
- 高效时段分布
- 不同提醒/建议方案的效果差异

## 11. 视觉与交互方向

### 11.1 视觉关键词

- 柔和
- 轻盈
- 灵动
- 愉悦
- 中文主导

### 11.2 视觉系统建议

- 色板以奶油白、鼠尾草绿、雾蓝、浅金为主
- 页面结构避免厚重后台感，强调呼吸感与层次
- 采用柔和渐变、半透明层和轻投影
- 数据图表风格圆润、轻量、有动势

### 11.3 动效原则

首页：

- 极轻的漂浮光斑
- 轻视差背景
- 页面内容渐入

驾驶舱：

- 卡片缓入
- 关键数值上升动画
- 图表“生长”式进入

行动完成时：

- 柔和粒子
- 成长值递增
- 非夸张的庆祝反馈

周复盘：

- 切换时如翻开成长手账
- 洞察卡片按节奏出现

### 11.4 需要避免的问题

- 变成普通任务列表工具
- 为了炫技加入大量影响效率的动画
- 英文文案过多，破坏中文用户体验
- 图表过于冷硬、后台化

## 12. 技术架构

### 12.1 总体方案

优先采用单栈全栈方案，以控制时间和复杂度：

- 前端：Next.js + TypeScript + Tailwind CSS + Motion
- 服务端：Next.js Route Handlers / Server Actions
- 数据库：Cloudflare D1
- 部署：Cloudflare Workers
- AI：规则模板优先，Workers AI 作为可插拔增强

说明：

- Next.js 部署到 Cloudflare Workers 时，默认按 OpenNext for Cloudflare 路径适配
### 12.2 采用该方案的原因

- 对作品集来说，单栈方案更稳，交付效率更高
- 可同时满足前端表现力、后端接口、数据库和部署
- 相比多平台拼接，更利于控制免费额度和大陆访问稳定性

### 12.3 分层结构

展示层：

- 页面、组件、图表、动效

业务层：

- 目标管理、任务流转、复盘生成、勋章逻辑

AI 层：

- 目标拆解、周总结、重规划建议

数据层：

- 用户、目标、任务、复盘、事件

分析层：

- 指标聚合、漏斗计算、分群分析

## 13. 中国大陆访问与免费约束

### 13.1 访问策略

产品面向中国大陆用户，因此在部署策略上优先满足：

- 海外部署但尽量对大陆用户友好
- 尽量减少跨平台多跳调用
- 避免必须依赖付费海外 SaaS 才能跑通

基于当前官方文档和工程可行性判断，第一版优先采用 Cloudflare 单平台部署路径。

说明：

- 该方案并不等于“中国大陆境内部署”
- 如果未来要做严格的大陆境内加速和合规，会涉及 ICP 等额外要求，不属于当前学生项目范围

### 13.2 免费优先原则

选型必须优先满足：

- 有免费计划或免费额度
- 额度足以支撑作品集级 Demo
- 失败时可回退，不因单一外部服务不可用而整体失效

这也是 AI 层采用“规则模板 + AI 增强”双轨设计的原因。

### 14.1 第一版必须完成

- 目标详情页

## 15. 异常与兜底设计

### 15.1 AI 调用失败

- 回退到规则模板生成基础计划或建议
- 页面不给空白状态

### 15.2 新用户没有历史数据

- 展示鼓励式空状态
- 引导完成第一个行动

### 15.3 任务过多或计划过载

- 主动建议压缩成 3 个关键动作

### 15.4 用户连续掉线

- 使用温和提醒和重新开始机制
- 避免高压式“失败”语言

## 16. 验证与测试要求

至少验证以下内容：

- 目标创建到复盘的主流程可走通
- 任务完成和事件埋点口径一致
- 周复盘页面可基于结构化数据生成稳定内容
- 中文内容在桌面端和移动端都具备良好可读性
- 首页和关键页的动效不会显著影响性能

## 18. 后续迭代方向

如第一版完成质量较高，可考虑：

- 提醒文案实验
- 个性化目标推荐
- 不同拖延类型的差异化干预
- 更细腻的成长报告表达
- 与学习/求职场景的轻扩展

## 19. 成功标准

GrowthPilot 第一版成功的标准不是功能越多越好，而是同时满足：

- 一眼看上去有品牌感和完成度
- 用户闭环完整且逻辑清楚
- 中文体验自然
- 数据后台足够说明 PM 思维
- 技术选型稳、免费可用、可部署
- 后续有明确迭代空间
